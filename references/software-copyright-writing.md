# 软件著作权（软著）登记材料撰写规范

本文件是 patent-drafting 技能 **§SC 软著分支** 的完整执行规范与"脑子"。命中"写软著/软件著作权/源代码材料/软著说明书"意图时通读本文，按其流水线执行。**软著是版权登记、无实质审查**——不跑查新/客体/创造性，只保证材料**真实、合规、自洽**。

> 法规与口径以《计算机软件保护条例》《计算机软件著作权登记办法》及中国版权保护中心（CPCC, ccopyright.com.cn）现行须知为准。本文格式硬规为通行口径（已核实，见下）；以 CPCC 登记系统现行表为最终准。

> ## ⚠️ 合规红线（2026-03-15 新规，每次必读、交付必告知）
>
> 中国版权保护中心自 **2026-03-15** 起启用新版《计算机软件著作权登记申请表》，建立"诚信制度"。已核实要点（来源：新浪财经 2026-03-17、软件行业协会转发官方《通知》）：
> - 签章页申请人须**手写并实名承诺**：**"本软件确系人的独立开发，未使用 AI 开发编写代码、撰写文档或生成登记申请材料"**；
> - 失实/欺骗 → **列入版权登记失信名单及个人征信记录**，并承担法律责任；
> - **经办人须签名并附身份证号**（实名核验）；
> - 主要功能描述字数由 200 字以内提升到 **500–1300 字**。
>
> **本技能的产出（说明书、申请表信息、功能描述等）是 AI 生成的"草稿/素材"。** 直接拿去签上述承诺将构成不实承诺、踩失信与征信红线。因此：
> - **SC-3 生成后、SC-4 交付时必须明确告知用户**：本输出仅为草稿，**不建议直接提交**，用户须**自行实质修改、核对并对真实性负责**后方可使用；承诺与征信风险由用户知情自负。
> - 源代码 60 页材料是对**用户自己编写的代码**做机械排版（非"AI 编写代码"），相对安全；说明书/申请材料文字属"撰写文档/生成申请材料"范畴，风险最高，务必让用户重写到可真实作承诺。
> - 不得把本技能宣传或使用为"一键生成可直接签字提交的软著材料"。

## 0. 软著基础（撰写前必读）

- **登记机关**：中国版权保护中心（CPCC），非国家知识产权局。保护的是**代码这一"表达"**，不保护思想/算法本身。
- **三样核心材料**：①申请表（线上填报）②软件鉴别材料 = **源程序 + 文档（说明书）** ③证明文件（身份证/营业执照/事业单位法人证书，合作或委托开发另附协议）。
- **无实质审查**：只做形式审查，不查新颖性/创造性。故本分支**绝不**做专利那套查新与客体论证。
- **费用/周期（通行口径，以官网为准）**：基础登记自 2017 年起免费；正常约 2–3 个月，有付费加急。电子证书与纸质同等效力。
- **诚实红线**：申请表所写代码行数与源代码文档页数必须**自洽**（写"五万行"却只交 50 页源码会被打回补正）；说明书/源码**不得出现真实用户数据、真实公司名、他名软件**。
- **2026-03 新规红线（见顶部⚠️）**：申请表须手写实名承诺"未使用 AI 撰写文档/生成申请材料"，违则挂征信。本技能产出一律按"草稿/素材"交付，**交付时必告知用户须自行实质修改、自负真实性与承诺**。

### SC-2 读码梳理（Agent 驱动 + 每栈启发式，见 §4）
先用 Glob/Grep 识别技术栈与入口，**只精读**界面定义文件、入口、事件回调，其余采样。产出**统一中间稿**：

- `软件简介`（面向什么需求、底层方法/框架、用户怎么操作→产出什么价值）
- `基本功能清单`（逐条，可回指代码）
- `运行环境`：**软件依赖**（框架/解释器/运行库版本）从依赖文件(`requirements.txt`/`package.json`/`*.csproj` 等)、README、安装脚本得出；**硬件配置（内存/CPU/分辨率/操作系统版本）源码里没有、推不出 → 问用户或标"待确认"，绝不凭空写"2G 内存/I5/Windows 7"等示例值**。
- `安装/分发形态`（**决定"二、软件安装说明"怎么写，须据实、不凭源码臆测**）：检测打包/安装产物——Inno Setup `*.iss`、NSIS `*.nsi`、WiX `*.wxs`、Electron `package.json` 的 `build`/electron-builder/`forge.config.*`、PyInstaller `*.spec`/cx_Freeze、Nuitka、MATLAB Compiler `*.prj`(需 MATLAB Runtime/MCR)、Qt `windeployqt`/Installer Framework、`Dockerfile`、`dist/`/`release/` 里的 `setup.exe`/`*.msi`、Makefile/CMake 的 install 目标、README 安装段。**据此判定**：有安装包→写安装步骤；免安装→解压即运行；仅能从源码跑→写依赖与启动方式。**检测不到或不确定 → 回到 SC-1 问用户，不要默认"运行 setup.exe"**。
- 有 GUI：`界面区块树`（主界面分哪些区块 → 各区块控件 → 控件标签与含义/取值）+ `逐步操作流程`。**操作流程要写"一条完整任务的正常主线(happy path)"，不是平铺所有回调**——按 **§4.5「操作逻辑还原」** 的"抽取 → 功能-顺序确认表 → 装配"流程做，主线与"其他功能"分开，顺序不确定就问用户。
- 无 GUI：`模块-函数-数据流`（模块划分、各模块输入输出、关键算法、调用关系）

**只写代码里真实存在的**；推断处标注"据…代码推断"。

## 2. 格式硬规（CPCC 通行口径）

### 2.1 源代码鉴别材料
- 提交**前 30 页 + 后 30 页 = 60 页**；不足 60 页**全部提交**。
- **每页 ≥50 行**（最后一页除外）；**无空行**（凑数注释可保留但别整页空白/纯括号）。
- 启动/入口代码放**文档开头**；**最后一页是程序自然结尾**（别停在写到一半处）。
- 页眉：**软件全称 + 版本号**，与申请表**字字一致**；页码在**右上角**（`第X页 共Y页`）。
- 等宽字体（Consolas/Courier New）、A4、单面打印。

### 2.2 文档（说明书）鉴别材料
- 同样前 30 + 后 30，不足 60 页全交（说明书一般不超 60 页，多为全交）。
- **每页 ≥30 行**；**有图的页不限行数**。
- 页眉：软件全称 + 版本号（与申请表一致）；右上页码。
- **不得含真实用户数据、真实公司名、他名软件**。

### 2.3 命名
- 全称建议以"系统/平台/软件/工具/APP"结尾；**简称不超出全称范围**；首次申请版本号填 `V1.0`。
- 名称在**申请表、源代码页眉、说明书页眉**三处字字一致。

## 3. 说明书文风（复刻工程软件房型，对齐样例）

页眉（两类通用）：`页码 + 软件全称 + 版本号`，每页重复；右上页码。骨架与就地范式见 `references/sc-templates/操作说明书.md` 与 `设计说明书.md`。要点：

### 3.1 操作说明书（有 GUI，默认）
- **一、简介**：`1.1 编写目的`——一句话"…是面向{需求}而开发的一套软件系统"，点出底层方法/框架 + 用户操作 → 产出价值；`1.2 基本功能`——`（1）（2）（3）…`逐条枚举（导入/计算/保存/导出/可视化等），每条对应真实功能。
- **二、软件安装说明**：`2.1 推荐系统配置环境`（操作系统、硬件、软件依赖如 MATLAB/.NET/运行库）；`2.2 软件安装方法`——**按 SC-2 测得的分发/安装形态据实写，不要默认 setup.exe**：①有安装包→"运行 setup.exe/.msi 安装，按向导完成"（含所需运行时安装，如 MATLAB Runtime/.NET）；②免安装→"解压压缩包后双击 XX.exe 运行"；③从源码运行→"安装依赖(如 `pip install -r requirements.txt`／`npm install`)后，执行 XX 启动"。**形态不确定就先问用户**。
- **三、软件使用说明**：`3.1 软件界面基本介绍`——主界面分为哪几个区块（如图1），逐区块解释每个控件/参数的物理含义与取值约束；`3.2 软件使用基本流程`——`第一步…如图2`、`第二步…如图3`，**只写一条完整任务的正常操作主线**(happy path)，按功能逻辑而非代码罗列顺序，每步说明输入、操作、产出；异常/急停/重置/可选高级等**非主线操作另设"3.3 其他功能说明"或并入界面介绍**，不混入编号步骤。顺序拿不准先让用户确认。

### 3.2 设计说明书（无 GUI / 算法后台）
一、简介（编写目的/基本功能）→ 二、运行环境 → 三、总体设计（系统结构图，mermaid）→ 四、模块设计（各模块功能、输入输出、关键算法，流程图用 mermaid）→ 五、数据结构与接口设计 → 六、出错处理设计。

### 3.3 通用纪律
- 功能、参数含义、操作步骤**逐条回指代码证据**，不夸大、不杜撰未实现的功能。
- **按功能逻辑组织，不照搬代码结构**：操作流程写正常主线，非主线操作单列；读码得到的回调清单是素材不是流程，需重排成人能照做的使用顺序，不确定就问用户。
- 图文一致：正文"如图X"与图题、图序一一对应；图中不出现正文未述的界面。
- 篇幅以说清功能为准；图文并茂（操作说明书尤其要有界面截图）。

## 4. 四栈读码启发式（SC-2 的"怎么读"）

目标：用最少 token 定位**界面定义文件 + 入口 + 事件回调**，还原界面区块树与操作流程；无 GUI 则还原模块-函数-数据流。先 Glob 看目录与文件类型判定主栈，再 Grep 关键模式。

### 4.1 MATLAB（App Designer / GUIDE）
- **判定**：有 `.mlapp`（App Designer）或 `.fig` + 同名 `.m`（GUIDE）。
- **App Designer**：`.mlapp` 是 zip 包——可解压读其中的组件/布局 XML（`matlab/document.xml` 等）得控件树；代码逻辑在内嵌 `.m`/`StartupFcn`/各回调。
- **GUIDE**：`.fig` 为二进制（不直接读），看同名 `.m`：grep `function .*_Callback`（每个控件回调）、`uicontrol`/`uibutton`/`uitable`、`handles.`/`app.` 属性，按回调名还原控件与操作。
- **入口**：`StartupFcn`、主函数、或 `app.UIFigure` 创建处。

### 4.2 Qt / C++（含 PyQt/PySide）
- **判定**：`.pro`/`CMakeLists.txt` + `.ui`/`.qml`/`.cpp`，或 PyQt 的 `.ui` + `.py`。
- **界面**：`.ui` 是 XML——直接读 `<widget class="..." name="...">` 与 `<property name="text">` 得控件层级、名称、显示文案；`.qml` 读组件树。
- **事件**：grep `connect(` / `Q_SLOT`/`slots:`（C++）、`.clicked.connect(`/`.triggered.connect(`（PyQt）还原"控件→槽函数"的操作流。
- **入口**：`main()` / `QApplication` / `loadUi`。

### 4.3 Python GUI（Tkinter / wxPython / PyQt）
- **判定**：import `tkinter`/`wx`/`PyQt`/`PySide`。
- **界面**：grep 控件构造（`Button(`/`Entry(`/`Label(`/`Frame(`…）、布局（`.pack(`/`.grid(`/`.place(`）还原区块。
- **事件**：`command=`（Tkinter）、`.bind(`、`.connect(` 还原回调。
- **入口**：`if __name__ == "__main__":` / `mainloop()` / `app.exec_()`。

### 4.4 Web / Electron
- **判定**：`package.json`（看 `dependencies`：react/vue/electron）。
- **界面**：组件树（`.jsx/.tsx/.vue`）、路由表（react-router/vue-router）、模板中的按钮文案；DOM 结构。
- **事件**：`onClick=`/`@click=`/`addEventListener` 还原操作流。
- **入口**：`src/main`/`index`、Electron 的 `main.js`（主进程）+ 渲染进程页面。
- **自动截图**：仅本栈 `sc_ui_capture.py` 用 playwright 较可行（见 §5 Tier 2）。

### 4.5 混合栈 / 找不到入口
- 混合栈（如 Qt 前端 + Python/C++ 算法后端）：**以主界面所在栈为主**还原操作流程，其余作为"后端模块"在说明书中简述。
- 找不到入口/界面定义：**请用户指定**入口文件或 GUI 框架，不硬猜。

## 4.5 操作逻辑还原（抽取 → 功能-顺序确认表 → 装配）

静态读源码只能得到"有哪些功能/回调"，**推不准"怎么用(顺序与主次)"**。故操作流程不直接由读码成文，而走"**抽取 → 分块 → 逐功能确认 → 装配**"，把流程锚定到最懂软件的作者。

### 4.5.1 抽取（先有料，再推断）
两路并行：
- **A 读码（有什么功能）**：四栈启发式（§4）拿到控件/回调/菜单项，每个候选功能记 `名称、触发控件、对应回调、所在区块、代码位置`。
- **B 挖现有制品（怎么用、什么顺序——证据强于代码）**，按可信度从高到低，Glob 命中即精读、命不中即跳过（省 token）：
  1. **E2E/集成测试**（Playwright/Selenium/pytest-qt/Robot 等，常在测试目录）——点击序列常就是 happy path，**顺序证据最强**；
  2. **README / 帮助文档 / 教程 / 操作视频脚本**（`README*`、`docs/`）——含"第一步…"式说明；
  3. **菜单 / 工具栏 / Tab 顺序**——设计者排布隐含主次与先后；
  4. **示例数据 / demo 脚本 / 默认参数**（`examples/`、`demo*`）——示例跑法暗示典型流程；
  5. **代码内使能依赖**——按钮 enable/disable 前置、产出被下一步消费 → 数据依赖定序。
- 无任何制品时退回纯代码依赖，并**显式标注"顺序为代码推断、置信低"**。

### 4.5.2 功能-顺序确认表（结构化中间件，不入正文）
把候选功能汇成表（工作底稿）：

| 序号 | 功能名 | 推断含义(一句) | 证据(E2E/文档/菜单/示例/代码依赖) | 置信度 | 推断位置(主线第N步 / 其他功能 / 待定) |
|---|---|---|---|---|---|

推断规则：
- **主线** = 完成一次典型任务的最短正常路径（输入/导入 → 设参 → 运行 → 看结果 → 保存/导出）。
- **归"其他功能"**：异常/出错、急停/终止、清空/重置、撤销、帮助/关于、设置、可选高级、次级弹窗。
- **排序**优先用高可信证据（E2E > 文档 > 菜单 > 示例 > 代码依赖）；多源冲突时高可信胜，并把冲突项标"待定"待确认。
- **置信度**：多源一致=高；单源=中；仅代码依赖=低。低置信项一律进确认环节。

### 4.5.3 逐功能确认（确认粒度由运行时用户选）
先问使用者选哪种粒度（**默认 A**）：
- **A（先整体后局部，推荐）**：一次性展示整张确认表 → 用户自由文字纠正（改含义/改顺序/调主次/增删）→ **仅对低置信或被改动项再逐个追问** → 定稿表。
- **B（严格逐功能）**：按表逐个问"该功能含义是否为 X？属主线第 N 步还是其他功能？"逐一确认。
两种都产出**「确认后的功能-顺序表」**；**不得跳过确认直接成文**（除非用户明确弃权，此时全文标"未经确认、按推断"）。

### 4.5.4 装配
- 主线功能按确认顺序 → 写 `3.2 软件使用基本流程` 的第一步/第二步…（每步 输入→操作→产出 + "如图X"）。
- "其他功能" → 写 `3.3 其他功能说明`。
- `1.2 基本功能` 据此表枚举。确认表本身是工作底稿，**不写入正文**。

## 5. 截图三级兜底（操作说明书的界面图）

操作说明书绕不开界面截图。按以下降级链处理，**优先 Tier 1**：

- **Tier 1（默认，永远可用）**：读码生文。生成全部文字 + 每处"如图X"图题 + 在 md 中留**插图占位**（如 `![图1 软件主界面](占位-图1)`）。不依赖软件能跑。
- **Tier 2（自动，按栈）**：
  - **Web/Electron** → `sc_ui_capture.py`：playwright 起本地服务、按路由/操作流自动截关键界面，回填占位。最可行。
  - **桌面（Qt/Tk/MATLAB）** → best-effort：可在用户本机用 computer-use 起应用截图，**脆、需用户在场，默认不强上**；不可行即落 Tier 3。
- **Tier 3（引导补图）**：输出**截图清单**——逐图说明"图X = 哪个界面、在软件什么操作下出现、贴回 md 哪个占位"，docx 中保留占位框，由用户补真图后重出。

无 GUI 软件：直接转**设计说明书**，用 `mermaid_render.py` 出结构图/流程图代替截图，免此问题。

## 6. 合规自检清单（SC-4，内部执行，不写入正文）

## 7. 工具与命令（在 conda 环境 patent-drafting 中运行）

```bash
# 源代码 60 页材料（可多次 --src 指定文件夹；--dry-run 仅做合规自检、不出 docx）
conda run --no-capture-output -n patent-drafting python "$SKILL_DIR/tools/sc_source_doc.py" \
  --src <代码文件夹> [--src <另一个文件夹>] \
  --name "<软件全称>" --version "V1.0" \
  --out "<输出目录>/源代码.docx"

# 说明书：写成 md 后转 docx（复用现有工具）
conda run --no-capture-output -n patent-drafting python "$SKILL_DIR/tools/md_to_docx.py" --input <说明书>.md --output <说明书>.docx

# 设计说明书的结构图/流程图（mermaid → PNG，复用现有工具）
conda run --no-capture-output -n patent-drafting python "$SKILL_DIR/tools/mermaid_render.py" <图>.md

# Web/Electron 自动截图（Tier 2，可选；失败即降级）
conda run --no-capture-output -n patent-drafting python "$SKILL_DIR/tools/sc_ui_capture.py" --url http://localhost:3000 --out <截图目录>
```

环境缺失/脚本不可用 → 走兜底（Agent 手工读码生文 + 引导用户补源码材料），并标注。
